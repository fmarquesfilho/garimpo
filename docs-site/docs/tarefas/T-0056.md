## Descrição

A v3 da state machine funciona logicamente (294 testes passam), mas a UI
apresenta vários problemas visuais e de dados que degradam a experiência.
São bugs de binding e de contrato API↔frontend.

## Atualização 2026-07-12 (pós-Omnibox / T-0055)

A BuscaEngine (FSM headless) continua sólida e foi reusada sem alterações pelo
Omnibox — feedback do dono do produto: **"a engine looks good"**. As demais
partes da página estão "getting there".

⚠️ **PENDENTE — o workflow de lojas (store) precisa de refactor e correção.**
É o ponto fraco atual da página Descobrir. Sintomas/áreas a revisar:
  - Resolução/adição de loja (via `@loja` no omnibox e via link/ID) e o
    caminho `ADICIONAR_LOJA` (monitorada vs. resolver nova via Collector).
  - Nomes de loja com espaço (`Glory of Seoul`) não casam com token sem espaço
    (`@glory` casa por match parcial; `@gloryofseoul` não) — a UX de completar
    loja no omnibox contorna removendo o token e mostrando card, mas o modelo
    de dados/derivação de lojas monitoradas (`listarLojasMonitoradas` a partir
    de buscas salvas) e o escopo por loja (`shopIds` × marketplace) merecem
    um desenho mais consistente.
  - Estados `lojaResolvendo`/`lojaErro` e o fluxo de "loja nova por URL/ID".
Recomenda-se uma task dedicada de refactor do subsistema de lojas (escopo,
resolução, nomes, monitoramento) antes de novas features de busca.
